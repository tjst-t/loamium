スマートコマンド 編集モード案

commands/create-todo.md (frontmatter: loamium-command) を開いたときの編集 UI の方向性を 5 種類並べています。どれが使いやすいか確認してください。

ユーザー要件まとめ: GUI(フォーム)編集 + YAML 直接編集の両方を提供し、 YAML 直接編集では不正な定義は保存前にエラーを出す。 → この要件に最も近いのは V4 — タブ切替 です。

バリアント一覧

  1. V1 — コマンド対応モード(Markdown 維持)オーバーレイパネル / ソース維持
  2. V2 — フォーム編集(構造化・YAML 非表示)最も直感的 / YAML 非表示
  3. V3 — スプリット(YAML + ライブプレビュー)常時両面表示 / 幅が必要
  4. V4 — タブ切替(フォーム ⇄ YAML)★ユーザー要件に最近接 / 実装コスト中
  5. V5 — ミニマル(検証バナー + テスト実行のみ)最軽量 / フォームなし

V1 — コマンド対応モード(Markdown 維持)

通常 MD エディタ + トップパネルを重ねる。ソース true が維持される。フォームなし。

PROTO NOTE エディタ上部に薄いオーバーレイパネルを付ける方式。YAML ソースが正本のまま。 フォーム編集は無しのため、YAML を知らないユーザーには厳しい。 V5 と似るが、パース済みプレビューとテスト実行ボタンを加えた強化版。

V2 — フォーム編集(構造化・YAML 非表示)

YAML を完全に隠す設定フォーム。直感的だが柔軟性は低く、スキーマ全体の網羅が実装コスト高。

PROTO NOTE YAML は一切表示しない。すべてフォームで操作する。非エンジニアに優しいが、将来ステップ種別が増えるたびに フォーム UI の更新が必要。エラー前保存ブロックは下段の invalid 例で示す。

V3 — スプリット(YAML + ライブプレビュー)

左ペイン YAML 編集 / 右ペインリアルタイム検証。画面幅が必要。パワーユーザー向け。

PROTO NOTE 常時 2 ペイン表示。左でソースを編集すると右がリアルタイム更新。 モバイル非対応。横幅が狭いと使いにくい。エラー前保存ブロックは下段 invalid 例で示す。

V4 — タブ切替(フォーム ⇄ YAML) ★ユーザー要件最近接

フォームと YAML を同期しつつタブで切替。YAML 保存ブロックを明示。

PROTO NOTE ユーザーが要求した「GUI + YAML 両方」に最も近い。フォームタブで直感編集、 YAML タブで細かい制御。YAML タブでは不正な定義は保存前にエラーを表示し保存ボタンを無効化する。 両タブは常に同期している(フォームを変えると YAML が更新、YAML を変えるとフォームに反映)。

V5 — ミニマル(検証バナー + テスト実行のみ)

通常エディタ + バナー 1 行のみ。最軽量。フォームなし。V1 のサブセット。

PROTO NOTE V1 のさらに軽量版。コマンドオーバーレイパネルを持たず、エディタヘッダに 1 行バナーを付けるだけ。 実装コスト最小。エラーはバナーに出るが、テスト実行ボタン + バナー以外は追加しない。 V1 と比較対象として提示。

確認したいオープン質問

  1. V4 を選ぶ場合: タブ切替時の同期はどちらが「正本」? フォーム操作 → YAML 生成、YAML 操作 → フォームに反映、の双方向同期が必要? または YAML が常に正本で、フォームは読み取り専用プレビューでよい?
  2. 保存トリガー: YAML タブで保存ボタンを押す vs. CodeMirror のオートセーブで都度保存? (後者は「エラーでも保存してしまう」ため V4/V3 ではブロックが必要になる)
  3. フォーム ↔ YAML の同期タイミング: リアルタイム(keystroke)? タブを切り替えたとき? または手動「適用」ボタン?
  4. ステップ種別の増減: 将来 agent-run が追加された際、フォーム(V2/V4)は自動的に対応できるか? それとも都度フォームUIを更新する必要があるか?
  5. ミニマル(V5)とする場合: YAML ソースで保存をブロックしない設計にするか? (現在のノート API は何でも書けるため、エラー定義は「実行時に 4xx で返す」でもよい)
