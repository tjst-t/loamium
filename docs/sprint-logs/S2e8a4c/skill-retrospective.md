## 1. Sa6c3b0 モバイル Playwright テストが初回 5 件失敗 → 修正ラウンド

- **事象**: 実装第1版で mobile-bottom-nav 非表示 / スクリム未クローズ / Agent シート開閉不能の 5 テストが fail。原因は (a) CSS ソース順カスケードで `.mobile-bottom-nav{display:none}` がメディアクエリの `display:flex` を上書き、(b) スクリム幾何、(c) シートの pointer-events。
- **分類**: **task-local**(実装 CSS バグ)。スプリント自身の Playwright テストが検知し、修正ラウンドで解消。ゲートは機能した。
- **提案/見送り**: SKILL 変更なし(見送り)。ただし「CSS ソース順で後方の global 規則がメディアクエリを上書きする」は再発しやすいクラス。→ **軽微な提案**: 実装エージェント prompt に「レスポンシブ CSS はメディアクエリを基底規則より後方に置くか詳細度で勝たせる」を1行追加(任意)。

## 2. ネスト `<button>` による不正 HTML → flex レイアウト崩壊(プロトタイプ Dataview)

- **事象**: プロトタイプの行が `<button>` 内に `<button>` をネストし、パーサが外側を閉じて flex コンテナが空になり整列崩壊。ユーザーがスクショで指摘 → 11 行を `<div role=button>` に変換して解消。
- **分類**: **SKILL/harness 欠陥候補**。プロトタイプ生成エージェントが繰り返し陥りうる罠(実装 UI でも同じ)。
- **提案(SKILL diff)**: `sprint-prototype.md` と実装エージェント prompt に「インタラクティブ要素をネストしない(`<button>` 内に `<button>`/`<a>` を置かない。行全体をクリック可能にするなら `<div role="button" tabindex="0">`)」を明記。加えて簡易 lint(`grep` で `<button>…<button>` を検出)を prototype/実装の検証に追加できる。→ **PO へ提案**。

## 3. Se3b7a2 データモデルの反転(alt-checkbox → インラインフィールド)+ プロトタイプ 4 レビューラウンド

- **事象**: 初回エラボレーションは status を alt-checkbox(`[/]`/`[!]`)で設計。レビューでユーザーが「単純タスクに過剰・前知識が要る」と指摘し、Dataview インラインフィールド + settings.yaml 語彙へ反転(ADR-0029)。
- **分類**: **task-local(想定内の設計反復)**。プロトタイプレビューゲートは「実装前に load-bearing な設計判断をユーザーと確定する」ためにあり、まさにその通り機能した(実装後の手戻りゼロ)。
- **提案/見送り**: SKILL 変更なし(見送り)。ゲートが手戻りを防いだ好例。強いて言えばエラボレーション時に「単純ケースの UX」を明示検討する観点を持てたが、レビューで吸収済み。

## 4. プロトタイプ生成エージェントの index.html / TESTIDS.md 同時編集(競合リスク)

- **事象**: 並列プロトタイプエージェントが共有ファイル(index.html / TESTIDS.md)を同時編集する設計だと clobber リスク。初回バッチでは各自追記で運良く成立したが、再改訂ラウンドでは main ループが編集を集約して回避した。
- **分類**: **SKILL/harness 欠陥候補**(並列 fan-out の共有ファイル競合)。
- **提案(SKILL diff)**: `sprint-prototype.md` に「並列プロトタイプエージェントは共有インデックス(index.html)/契約(TESTIDS.md)を編集しない。各自は自分の画面 HTML と testid デルタを**返す**だけとし、main ループが集約マージする」を明記。→ **PO へ提案**。

## 5. 大規模 GUI スプリントの単一エージェント実装がバジェット切れ(Sa6c3b0)/ 層分割が有効(Se3b7a2)

- **事象**: Sa6c3b0(6 ストーリー)を単一エージェントに任せたら検証途中で要約なく返却(バジェット切れ)、修正ラウンドを要した。Se3b7a2(8 ストーリー)は foundation/UI の 2 層に分割したエージェントで完走。
- **分類**: **SKILL/harness 欠陥候補**(1 エージェントのスコープ上限に対する分割方針が未明文)。
- **提案(SKILL diff)**: autopilot Sprint loop 実装フェーズに「ファイル共有のため直列化する大規模スプリント(概ね ≥5 ストーリー)は 1 エージェントに丸投げせず、依存レイヤ(shared/server → ui など)で 2 エージェントに直列分割し、各エージェントに検証+コミットを課す」を明記。→ **PO へ提案**。
